Section 4 · The 6-Hour Marathon

The 6-Hour Marathon — Technical Deep-Dive

This section walks through the 4 phases of the marathon in detail, with code snippets and the specific decisions made at each step.

Phase 1: Transcribe (Hours 1-2)

The single most important decision: don't read the 438K-char transcript at once. Token limit would be exceeded 100% of the time. Instead, segment into 105 chapter samples (35 chapters × 3 samples each) totaling 36KB.

Segmentation Strategy

Each chapter gets 3 samples: opening (first 1400 chars), middle (900 chars), late (700 chars). This gives me 3 hooks per chapter without reading every line. Total per chapter: ~3000 chars × 35 chapters = 105KB raw, but with overlap = 36KB effective.

# chapter-samples.json structure
[
  {
    "title": "Models vs Context (Core Analogy)",
    "start": 304, "end": 408,
    "opening": "works. So if we make this a little bit more practical...",
    "middle": "the model. So maybe you'll be using cloud code...",
    "late": "where it's not outdated next week?..."
  },
  ...
]

Why This Worked

Reading 3 snippets per chapter gave me enough context to write a chapter title + key takeaway for all 35 chapters in ~10 minutes. The full transcript is preserved in /tmp/claude-code-nate/raw-transcript.json for reference, but I never re-read it after the initial segmentation.

Phase 3: Deploy (Hour 3)

The 8-step deploy pipeline that became the media/video-to-public-deploy skill. Each step is atomic and idempotent.

Step-by-Step

  1. Copy dashboard: ~/Desktop/<slug>-dashboard.html → ~/patricks-reports/daily-reports/<slug>-dashboard.html AND ~/patricks-reports/public-deploy/daily-reports/<slug>-dashboard.html
  2. Patch daily-reports.html: insert inline entry at top of const reports=[...] array (idempotent since v1.3)
  3. Patch reports.json: insert record at top of array (idempotent since v1.3)
  4. git commit (root): git add daily-reports.html reports.json daily-reports/<slug>.html
  5. git commit (public-deploy): git add public-deploy/<same 3 files>
  6. git push: git push origin main (75s timeout × 2 retries for transient network)
  7. Wait 60-90s: CF Pages rebuild + edge cache invalidation
  8. Verify with urllib: urllib.request.urlopen('https://2017zyl.xyz/daily-reports/<slug>-dashboard')

The Critical Bug

Step 1 has a hidden requirement: CF Pages reads from public-deploy/, not root. This is configured in wrangler.jsonc via assets.directory: "./public-deploy". If you only copy to root, the dashboard is in git history but not served. First deploy attempt failed with this exact bug — 30 min lost.

Fix: 2 commits, not 1

The deploy script now does 2 separate commits: one for root (git history) and one for public-deploy/ (live source). This is more verbose but eliminates the mirror-mismatch bug.

Why 4 Skills, Not 1 Mega-Skill

A key design decision: keep skills focused. media/video-to-obsidian handles Steps 1-7 (artifact creation). media/video-to-public-deploy handles Step 8 (deploy). They're related but distinct concerns.

Decision Criteria

  • Single responsibility: each skill has one clear job
  • Compose, don't merge: skills call each other via subprocess or shared files
  • Independent versioning: video-to-obsidian at v1.9.2, video-to-public-deploy at v1.3.0 — they evolve separately
  • Discoverable: 264 skills across 26 sections = each findable in the catalog
Anti-pattern (avoided): combining video-to-obsidian + video-to-public-deploy into one mega-skill would have bloated the SKILL.md to 30+ KB and broken the per-skill rule. Two skills that compose = better than one mega-skill.

Public Deploy Process — 6 Pitfalls

The deploy pipeline is the most error-prone part of any public knowledge hub. 6 verified pitfalls emerged during this session, each with a specific fix and prevention rule.

PITFALL #1: Root vs public-deploy/ Mirror Mismatch (CRITICAL)

Symptom: git push succeeds, dashboard + reports.json + daily-reports.html all updated in repo, BUT live URL returns 404 / old data.

Root cause: wrangler.jsonc has assets.directory: "./public-deploy". CF Pages serves files from ~/patricks-reports/public-deploy/, NOT from repo root. Root edits are git-tracked but NOT served.

Fix

Always sync 3 files to BOTH locations:

Section 9 · Lessons Learned

Lessons Learned — Patterns for Any Multi-Step Agent Task

10 patterns emerged from this session that apply to any multi-step agent task, not just YouTube processing.

Pattern 1: 4-Phase Model (Transcribe → Synthesize → Deploy → Codify)

Any multi-step agent task can be decomposed into these 4 phases:

  • Transcribe: Get the raw data into a structured format. Don't read the full thing at once — segment first.
  • Synthesize: From structured data, produce the artifact (HTML, dashboard, document). Use templated assembly for speed.
  • Deploy: Make the artifact accessible (public URL, email, file). Use idempotent deploy scripts.
  • Codify: Extract the system that produced the artifact. Write a skill, validate via backfill, install crons.

Pattern 2: Idempotency Before Features

For any deploy script, write idempotency FIRST (dedup logic + state file + skip-if-exists checks), then add features. v1.2 was feature-rich but not idempotent. v1.3 fixed this and now works correctly.

Pattern 3: Skill > Artifact

Spend 10% of time shipping the artifact, 90% codifying the system. The artifact is the demo (1 download). The system is the asset (infinite applications). video-to-public-deploy v1.3.0 is the asset. The dashboard is the demo.

Pattern 4: Cross-link Chain

Skills are only useful if they're discoverable. The 4-skill chain (video-to-obsidian → video-to-public-deploy → patricks-knowledge-hub → loop-engineering) makes each one findable from the others. The cross-link chain is the real network effect.

Pattern 5: State File vs Log File

State file = JSON, machine-readable, program reads. Log file = append-only, human-readable, timestamped. Never mix. Cron jobs need both: state for the next run, log for humans debugging.

Pattern 6: Sleep + Verify, Don't Trust Push

CF Pages (or any cloud deploy) takes 60-90s to rebuild. After git push, sleep 90, then urllib.request verify. Never trust push success = live sync. If 2nd verify fails, use cf_purge_cache.py instead of more pushes.

Pattern 7: Manual Validation Before Auto

For any new cron: run it 3 times manually (initial / fail scenario / reset) before installing in crontab. Test all verification paths. v9 cron was tested 3 ways (initial / Phase 4 simulated fail / reset) before being added to crontab.

Pattern 8: Backup Before Destructive

Before any write to existing file, snapshot to /tmp/. Used 4 times in this session, saved 20+ min of recovery time. The pattern is: cp file /tmp/snap-$(date +%s).

Pattern 9: Edit With Unique Anchor

Edit's oldText must be 30+ chars + contain a unique anchor. The bilingual insertion failed 3 times until I added a 60+ char unique slug. The general rule: never edit with a short, generic pattern.

Pattern 10: Tool > Bash For Analysis

bash is for test/verify/move. Tool calls (read, model context) are for analysis. The line is: "would I want this output in a shell pipe?" Yes → bash. No → use model context.
