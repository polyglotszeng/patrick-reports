🔥 核心论点

Claude Fable 5（俗称 Fable）作为 Anthropic 最强模型，在被美国政府禁用 18 天后重新上线 Claude 订阅，但有两个硬限制： (1) 仅到 7/7；(2) 每周用量超过 50% 后只能改用按量付费 API。Peter Yang 给出了他认为最值得花 Fable 的 5 个 use case，核心论点 = "用便宜模型 (Opus / GPT) 准备 + 用 Fable 做高杠杆决策"。

核心论点分层（5 个使用场景的杠杆由高到低）：

  1. 找 Fable-worthy 任务：让 Fable 自己审视你的项目 + memory → 列出它认为值得跑的高杠杆任务清单 [00:01:30]
  2. 获取长期人生 / 商业建议：喂 plan doc + 接好 MCPs/APIs → Fable 写一份 3-12 个月级别的方向 1-pager [00:04:13]
  3. 让项目 ship-ready：Fable spawn 多个 agent 全量扫代码库找 bug，比 Opus/GPT 找出 3×+ 的关键 bug [00:09:16]
  4. 规划下个大项目：用 Fable 写详细的 HTML/Markdown plan（含架构图、决策点、风险、验收标准），再交给 Opus/GPT 执行 [00:12:10]
  5. 重构大型代码 / Skill 系统：把个人 OS / Skill 库当作"大型 code base"，Fable 找出死代码 / 重复 / 风险 [00:15:54]
⚠️ 双重稀缺：Fable 同时被时间窗（7/7 截止）和使用量（50% weekly cap）双重限制。 Peter 明确建议：不要 "YOLO Fable" 全开 Ultra Code effort，否则 1-2 小时就烧光。默认 high effort + babysit。

🎯 Use Case 1 · 找 Fable-worthy 任务

时间：[00:01:30 - 00:04:13] · 核心动作：让 Fable 自己审计你的项目历史 + memory → 列出它认为"值得跑"的 Top 5 高杠杆任务。

Prompt 模板

"You are Fable 5, the most capable AI model available. Look through my projects and memory. List the top five tasks that require deep thinking that you think are generally worth running by you."

加上 Please do not share any confidential information because I'm making this video 防止敏感信息泄露。

Peter 给出的 5 个 Fable 推荐任务（基于他自己的项目）

  1. H2 strategy stress test — Fable 从 memory 里发现 Peter 刚离职 full-time 创业 + 有一批 advice skills → 推荐做战略压力测试
  2. Designing a new premium tier offer — 设计高客单价的旗舰 creator offer（pulls context from various sources）
  3. Content-to-conversion audit across every platform — 跨 newsletter / YouTube / shorts / social 全平台 conversion audit，pull data via APIs + MCPs
  4. Mine your archive into a content roadmap — 把 3 年 newsletter + 1 年 YouTube 全部 transcripts 喂进去，提炼 themes → repurpose 成 shorts
  5. Audit personal skill system — 审计 40+ 个 AI skills，找出 dead code / 重复 / 风险（Peter 说他会留到最后用 Fable 做这个）

共同模式

Fable basically wants to look at a large corpus of information data and extract insights from it and take action from it, right? Which is exactly what a state-of-the-art AI model is really good at.

Fable 推荐的 5 个任务都符合一个模式：大语料 → 提炼洞察 → 行动。这正是 SOTA 模型区别于便宜模型的根本差异。

Step 3 · The advisor skill（optional but powerful）

Peter 写了一个 /advisor skill，结构包括：

  • Use the skill whenever I'm stuck on a decision or asking for a gut check（trigger 描述）
  • Gathering life context — 通过 MCPs/API 拉数据
  • Escalating to the council — 3 个 persona 互相辩论：

🚀 Use Case 3 · 让项目 ship-ready

时间：[00:09:16 - 00:12:10] · 核心动作：Fable spawn 多个 agent 全量读 codebase + 找 bug + 修 bug。

Peter 的实战对象

Prompt 模板

"I'm about to ship this fitness app that lets users track their workouts. I want you to find everything wrong with it. Read the whole code base, look for real bugs, broken edge cases, anything that fall over in front of a user. List all of the things that you think need to be fixed and maintain a high quality bar."

  • Spawn 5 个 agent 并行扫代码

🗺️ Use Case 4 · 规划下个大项目

时间：[00:12:10 - 00:15:54] · 核心动作：用 Fable 写详细的 HTML/Markdown plan（含架构、决策、风险、验收标准），再交给 Opus/GPT 执行。

实战案例：Fork 健身 App，加 Nutrition Tracking Tab

Prompt 模板

"Hey, I want you to draft a detailed plan for a new nutrition tracking tab in this app. Please use searches online to find relevant resources, and then lay out a full plan, the key phases, the decisions we need to make, the risks, and the open questions that I need to answer. Flag anything that could sink the project if I get it wrong. Give me a plan clear enough that a cheaper or simpler model could execute step-by-step. Also, please write out the plan in HTML so that I can review it."

Fable 自动产出的 plan 包含

  1. Proposed experience design — Fable 自己画了 UI 草图（Apple 风格的 nutrition tab）
  2. Architecture — Supabase DB + 数据模型
  3. Key decisions to make：
    • Food data source：USDA FDC only（免费）vs 商业 API vs Open Food Facts
    • Day 定义 + cloud sync 策略 + log 节奏
  4. Risks + open questions

进阶：用 Lavish Editor（Kun Chen 的 GitHub HTML editor）做 plan review

Peter 用 Lavish Editor（朋友 Kun 做的 free GitHub HTML editor）打开 Fable 生成的 HTML plan：

  • 直接在 plan 上 comment（如 "use US FDC only"）
  • Comment 直接转给 Fable 继续迭代（像 Google Doc 一样）

进阶：用 /go 命令 + 验收标准

如果要 Fable 自己执行 plan（不交给 Opus/GPT），可以：

  • 给它一些目标图片做 design acceptance
  • 用 /go 让它循环到 reproduce 出目标 design
  • 代价：会快速烧光 limit
⚠️ Peter 的建议：用 Fable 做 plan，把 execution 交给 Opus/GPT。除非是非常关键的简单任务，不要让 Fable 自己实现。

💡 三条使用 Tips

Tip 1 · Prep with cheaper models

让 Opus / GPT 做准备（写 plan doc / 接 MCPs / 写 draft），Fable 做决策。

Don't use Fable for boring work like hooking up an MCP or API or like trying to get you draft a document.

反例：用 Fable 接 API = 浪费 limit。

Tip 2 · Plan with Fable, execute with another model

Fable 写出详细的 plan（含 design + architecture diagram），交给 Opus/GPT 执行。

Once you have a detailed plan, you can then pass this plan to another model that's cheaper to do the actual execution.

本质：Fable 是战略家，Opus/GPT 是工程师。

Tip 3 · Use lower effort + babysit

不要全开 Ultra Code effort！

  • 默认 high effort 即可，Ultra Code = 1-2 小时烧光
  • Babysit Fable：它会进 rat hole 循环烧 token
  • 监控：token 消耗速度 + 进度
Fable is really smart, but sometimes it can get into a rat hole and just keep looping on burning your tokens and you don't want that to happen since the tokens are so limited here on your subscription.

📐 Model-tier routing 建议（Patrick 视角）

  1. T0 (本地 4B) — 接 MCP / 写 draft / 跑 boilerplate → 0 cost
  2. T1 (Codex / GPT-5.5) — 实际 coding execution / 单元测试 → 低 cost
  3. T2 (Opus / GPT-5.5) — 详细 plan / 大语料分析 / prep 工作 → 中 cost
  4. T3 (Fable 5) — 战略决策 / bug hunting / 复杂规划 → 高 cost (稀缺)
关键 routing 规则：Fable 5 在 7/7 前是 有限资源，T2 → T3 的升级必须有明确的高杠杆场景。default = T2，Fable 只做"5 个 use case"。

🎓 主流学界 / 业界可能的反方观点（5+ 命名）

1. "Fable 找 bug 多 ≠ 找 bug 好" — 普通开发者

论点：Fable 找 12+ critical bugs，但其中很多可能是 false positive 或 stylistic issue。真正的 critical bug 数量可能跟 Opus/GPT 接近。

证据：Peter 没公开 bug 列表分类，无法验证 severity 分布。

Patrick 视角：⚠️ 部分警示 — 如果 Fable 报告大量 "minor" bug，会稀释 critical 信号。

2. "Plan doc 4 sections 是 ceremony 而非 leverage" — Pragmatic Engineering

论点：写 4 sections 的 plan doc 要 2-3 小时，可能比 Fable 跑 1 次还久。对低频任务不值得。

证据：Peter 自己承认 council skill 是 "very, very optional"。4 sections 的 prep 时间 vs Fable 输出时间可能不平衡。

Patrick 视角：✅ 同源警示 — plan doc 应该是 渐进式 而非一次性写完。

3. "MCP 接入 = 隐私/合规风险" — Security Researchers

论点：把 Mercury / Substack / Google Workspace 接 MCP 后，数据流过 Anthropic。即使 prompt 里写 "do not share confidential information"，data 在传输 / 存储层无法保证。

证据：Anthropic 数据保留政策的具体条款 + 是否对 Fable use case 启用 enterprise-grade privacy 是 unclear。

Patrick 视角：⚠️ 需 boundary check — patricks-vault-os 含 TELOS / 财务数据，接 MCP 需评估。

4. "Council 3-persona debate 质量不稳定" — Multi-Agent Researchers

论点：3 persona (customer / skeptic / operator) 互怼的结论质量取决于 prompt 设计。如果 persona 定义不清，可能只是 echo chamber。

证据：Peter 自己承认 council 是 "optional" + "fun to build"。学术上 multi-agent debate 的 quality ceiling 是 well-known。

Patrick 视角：⚠️ 同源警示 — 多 agent debate 是 research toy，不是 decision tool。

5. "架构 lock-in 风险" — Cloud Architect

论点：一旦接 MCP + plan doc 都按 Fable ecosystem 设计，你已经被锁定。换 model (e.g. GPT-6) 要重搭所有 integration。

证据：MCP 是 Anthropic 标准但 adoption 还 early。如果未来 OpenAI 主导 tool calling standard，迁移成本高。

Patrick 视角：⚠️ 同源警示 — 保持 model-agnostic 抽象层（如 Hermes Agent 的 plugin system）减少 lock-in。

6. "Opus / GPT-5.5 也能做这些" — Anthropic / OpenAI 自家用户

论点：Peter 自己承认 Opus/GPT 也能找 bug，只是数量少。差异是数量，不是类别。3-4 个 bug 可能是更 curated 的 signal。

证据：Peter 实测数据显示 12+ vs 3-4 = 3×。但 "12+ bug" 包含 minor，是否 "3× improvement on critical bugs" 需细分。

Patrick 视角：✅ 承认 — Fable 是 SOTA 但不是唯一选择，T2 模型已能 cover 80% 场景。

7. "Fable 5 被美国政府禁用 18 天" 的真相 — Regulatory Researchers

论点：Peter 说 "banned by the US government for 18 days" 但没给出 primary source。可能是 (a) export control (b) safety review (c) AI executive order。不同原因对应不同 timeline 和 policy implications。

证据：需查 Anthropic blog + FCC / Commerce Department filings + 2025-2026 AI regulatory timeline。

Patrick 视角：🔍 待验证 — 影响 Fable 未来可用性预判。
