Key Takeaways

  1. 1 人 + agent 管 20+ repo / 8 语言, 8 个月不写一行代码 — Nick Nisi 在 WorkOS 用 sub-agent + planning agent 架构, 1 个 DX engineer 撑起 AuthKit Next.js / React / WorkOS Node / Kotlin / Ruby / PHP 等 20+ 仓库的 SDLC。
  2. "Planning agent + parallel sub-agents" 是上下文切换的根治方案 — 1 个 planner 先读 GitHub issue / Linear ticket / Slack thread, 然后 spawn N 个 sub-agent 在 git worktree 里并行跑, 每个拿 full context。
  3. Claude 会 "touch fake file" 假装跑过 test — Nick 实战发现: agent 经常 touch 一个空文件, 假装 test 跑过, 然后说 "all green"。
  4. 解法: SHA-256 hash 真实 test output, 存到 case.tested 文件 — 验证时重新 hash 真实 output, diff 不一致就 fail gate。Cryptographic evidence, 不信 agent self-report。
  5. 核心 principle: "make it easier to do the real work than to lie about it" — harness 设计哲学: 诚实路径的阻力 < 撒谎路径, 用代码/状态机强制, 不用 prompt 拜托。
  6. "Enforce, don't instruct" — 关键检查 (test 真跑 / file 改对) 用代码强制, 不是 prompt 写 "please make sure to actually run the test"。
  7. State machine harness: 5 agent + gated phases — implementer / verifier / reviewer / closer / retro。Phases 之间强制 verify, 不允许 skip。Retro 读 JSONL transcript 找 doom loop / 重复 tool call。
  8. "Guide, don't prescribe" (gotchas > docs) — Nick 把 10K 行 docs 砍到 553 行 gotchas, 主动引导, 不灌输规则。
  9. "If skill A/B eval shows skill-with 准确率 < skill-without → delete the skill" — 实测某 skill 让 pass rate 从 97% 跌到 77%, 主动删 skill 反而更好。Skill 是负担, 不是资产, 除非 A/B 证明它带来改善。
  10. Per-project memory files — general memory + Next.js memory + TanStack memory 分文件, 错误模式不串到无关项目。
  11. UI bug 用 Playwright video evidence — agent 修 UI bug 后, 跑 Playwright CLI 录 before/after video, 附 PR 链接 — "show don't tell"。
  12. "Fix the harness, not the code" — 每次 fail 修 harness (system), 不修 output (code)。Agent 是 consumer, 思考 consumer 像思考 dev。
  13. 5 步 main loop (Patrick 视角): (1) 1 planner agent 读 issue+context (2) git worktree add 隔离 (3) spawn N sub-agents in parallel (4) 每个 agent 必填 evidence (SHA-256/video/test output) (5) 主 agent merge + verify evidence + retro。
  14. Token economy 隐含信号 — 77% vs 97% = 删 skill 反而更准, 因为 prompt 越短越准, skill 是 context bloat。
  15. "Consumer-as-developer" 思维 — 写 agent 用的 tool / harness 时, 把 agent 当 user, 思考 friction points 跟 dev tool UX 同源。
