1. Core Thesis

"Skill hell" is the new curse. Skill hell is where you have all of these skills available freely available that you can download, contribute to, you can figure out on your own, but you don't really know how the pieces all work together. You can't tell a good skill from a bad skill.

4-item checklist (the cure)

1. Trigger → 2. Structure → 3. Steering → 4. Pruning

Three predictions Matt escalates from

Tutorial Hell 2010s

Go to 100 tutorials, can't piece them together, stuck in loop.

Framework Hell 2015-2020

Every 10 minutes a new JS framework, learn the hot new thing always.

Skill Hell 2025+

Skills freely available but you don't know which are good. Individual + org level both affected.

Why this is a Patrick-level concern

Patrick has 19 active skills in ~/.hermes/skills/ + 100+ references + 7 vaults. This talk is a direct framework for the next audit / refactor wave. Matt also acknowledges his own guilt: "we have Matt PCO skills, which is my skills repo, which is one of the most popular engineering skill sets out there. And so I feel like I want to help the people who use my skills get out of skill hell."

2. Trigger — User vs Model-Invoked Skills

"Every time you add a model-invoked skill into your agent's environment, it increases what I'm going to call the context load on that agent... If you have 100 model-invoked skills, that's going to be 100 descriptions inside the context for your agent."

The core trade-off

Model-Invoked (autopilot)

Description ends up in agent context. Agent decides when to invoke.

Cost: 100 skills = 100 descriptions in context, tokens & decision fatigue per request.

Cost: Even when skill is perfect match, agent may choose not to follow it.

User-Invoked (manual)

disable_model_invocation: true. Description only visible to user.

Cost: 100 skills = 100 things user needs to keep in their head — cognitive load on the pilot.

Cost: But: zero unpredictability from agent decision.

Why Matt prefers user-invoked

"I much prefer to be in full control. That means I get to keep the context load on the agent as small as possible, but it does impose more of a cognitive load on me. So I need to understand the skills really deeply in order to get the most use out of them."

"Every time you have a model-invoked skill, it basically you get a cost in unpredictability because every time you have a context pointer pointing from one resource to another, the model may just choose not to follow it. You know, even if it's absolutely perfect for the task, it may just choose not to invoke the skill."

Decision framework for Patrick

ConcernChooseBecause
Token cost dominatesUser-invoked default100 skills = 100 descriptions gone
You need auto-firing for many tasksModel-invokede.g. planning, debugging, common ops
Skill is safety-criticalUser-invokedZero unpredictable firing
Skill is "always relevant"Model-invokedDescription IS the key
Patrick actual patternUser-invoked (manual cron, manual skill invocation)"Always know what's running"

3. Structure — Steps + Reference + Branches

Matt's mental model: a skill = 2 main units — Steps (the procedure) + Reference (supporting info). Three composition modes apply.

3 Composition Modes

All Steps simple proc

Linear procedure. No reference material. Example: "git commit and push" skill.

All Reference knowledge dump

Documentation lookup. No procedure. Example: TS-cheatsheet lookup skill.

Steps + Reference canonical

The standard form. Procedure embedded with supporting docs inline or behind context pointers.

Example: two-PRD skill (3 steps + 2 refs, single branch)

  1. Find relevant context
  2. Confirm test seams with user (HITL checkpoint)
  3. Write the PRD document

+ 2 reference items: "what is a test seam" + PRD template (both inline because single branch).

Branch decision — external context pointers

"Hide branching reference material behind context pointers. In other words, if you feel like your skill is going to be used in lots of different ways, then take the reference material that's relevant for those branches and hide them behind context pointers."

Example: domain-modeling skill

  • 2 branches: (a) update context.md glossary (b) create ADRs
  • Branch-specific reference material (ADR template + context.md template) → moved to separate files behind context pointers
  • skill.md stays slim — branches only include the routing logic

Why small skill.md

"Smaller skills are just easier to maintain, easier to audit, fewer words to think about. And every time you shave off a word, that is a token shaved. That multiple token shave from your skills cost."

4. Steering — Leading Words (Matt's Core Insight)

Matt: "I'm going to give you one technique that fixes this issue, which is the agent doesn't do what I want. This technique is the kind of main thing I want you to get from this talk."

"There are certain words that pack in a bunch of meaning into a very small space. These leading words are really powerful with agents because you put the leading word in the skill itself in the text and then the agent will repeat the leading word back to itself as part of its operations as part of its thinking tokens and as part of its output to you. And then because it's re-emphasizing that word and that word hopefully describes what you want from the agent that then goes and changes its behavior."

Concrete example — vertical slice

Don't do this

"Don't code layer by layer. Make sure that you create a small slice first and then go from there."

Vague, verbose, no terminology anchor.

Use a leading word

"vertical slice" — compact, established terminology, agent's prior activates.

"A vertical slice is a pretty well-known terminology in development. And so this will hopefully trigger the agent's prior."

Self-verification: read the reasoning trace

"The cool thing about this technique is you can know if it's worked because you say vertical slice in your skill and then you'll notice in the reasoning traces that it's saying, 'Okay, we're going to do this as a thin vertical slice.' then you should get better implementation plans."

Second lever: leg work isolation

Problem: "ask clarifying questions just doesn't ever do enough leg work. It sees that its ultimate goal is to create a plan. And so it just does a small amount of leg work with ask clarifying questions, ask you a couple of things, and then eagerly creates the plan."

Fix: split into separate skills, hide the future phase.

  • grill-with-docs — only the ask-clarifying-questions phase
  • two-PRD — only the write-PRD phase
  • After both: agent has done the full work, no shortcut possible.

"It's not always necessary to split skills into individual steps, but in particular cases where you really want an extra chunk of leg work. It really there's no technique like it. It works very, very well."

How to find leading words

"English is a pretty wide API in terms of different functions you can call, different things you can experiment with, and there are many leading word candidates out there, and agents are actually pretty good at helping you think of them."

5. Pruning — Sediment / No-Ops / DRY / Massive

5 failure modes Matt lists:

#ModeSymptomFix
1Massive skillskill.md > 20K charsFind underlying failure mode first (massive is symptom)
2DRY violationSame reference repeated in 2+ stepsSingle source of truth per concept
3SedimentMulti-contributor cruft accumulationRestructure or kill
4No-opsParagraph looks like it does something, doesn't influence behaviorDeletion test
5Agent-generated docsLLM-written skill that passes deletion test tautologicallyIndependent metric, not just deletion

The Sediment phenomenon (in Matt's words)

"Sediment is just a classic thing when people are working on the same set of docs really which is that everyone starts contributing to a shared markdown file. People add their own stuff. They don't feel brave enough to delete and modify anyone else's. And so you just end up with this huge amount of sediment with often irrelevant material for the skill, especially stuff that hasn't been laid out properly."

The deletion test (for no-ops)

"Let's imagine we have an implement skill and we have an entire paragraph of the skill that tells the agent to write a long detailed commit message. What would happen if you just deleted that paragraph? Well, the agent would probably still write a decent like long commit message."

The test: delete a paragraph. If outcome unchanged → the paragraph was a no-op → remove it.

How Matt keeps skills small

"People ask me a lot how I get my skills so small and it's just using these techniques, using deletion tests, using um making sure that I compact things into leading words, I don't have anything irrelevant in there, and I don't have any sediment."

6. Matt Pocock Skills vs Superpowers

Matt Pocock Skills

Style: User-invoked (manual control)

Cognitive load: High on pilot

Context load: Low on agent

Unpredictability: Zero — agent never auto-fires

Optimized for: Power users who know their skill set deeply

Look like: grill-with-docs, two-PRD, domain-modeling

Superpowers

Style: Model-invoked (autonomous)

Cognitive load: Low on pilot

Context load: High on agent — every description always present

Unpredictability: Real — agent may skip when relevant

Optimized for: Hands-off workflows, lower learning curve

Look like: Anthropic's flagship skill framework

What neither addresses

Multi-user / team composition

Both treat single pilot as the unit. Team / org skill composition un-discussed.

Skill versioning + evolution

No mention of backwards compat, deprecation, or migration paths.

Security: prompt injection surface

Description (in agent context) = read-then-act = potential injection vector. Neither framework discusses.
