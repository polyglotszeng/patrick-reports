⚡ Core thesis — five takeaways, in Zack's words

This interview is the most quotable Sierra-on-record session to date (Jun 2026). The recurring message: context engineering in a single brand-representing agent beats multi-agent decomposition in most cases, and the structural decisions (modular voice, separate PCI cluster, outcome pricing, ghostwriter + isomorphically-compiled journeys) all follow from that thesis.

"If you want a multi-agent system so that one team can work on one agent and one team can work on another agent, then you're shipping your org chart." [0:46:21]
"Showing agents everything they need to do the right thing, but nothing more." [0:40:01] — Sierra's definition of context engineering.
"Whenever you think the model's too dumb, the model's actually too smart." [0:41:43] — Context is usually the lever, not model choice.
"Agentic commerce will be bigger than e-commerce." [0:21:29] — Sierra is already on commission in sales outcomes.
"I'm kind of a monolith loyalist on that." [0:48:29] — Zack's stance on multi-agent: build the monolith first, decompose only with reason.

Five concrete takeaways

1. Monolith first, agents rarely

If you have well-managed context, the right decomposition is usually zero — Sierra agents are typically one agent per brand. Sub-agents or multi-agent only for truly separable jobs. [0:46:21] [0:48:09]

2. Voice is the proving ground

Majority of Sierra conversations are voice; 1-2s latency budget forces parallelization: think + listen + talk run concurrently, with multiple model calls (frontier + classifiers + voice-specialized). [0:18:00] [0:51:01]

3. PCI / payments = separate cluster

Payment info never reaches an external LLM. PCI DSS Level 1 certified by a QSA — "the only voice payments platform at launch where you don't need to transfer to another platform". [0:22:09] [0:21:44]

4. Memory is a first-class primitive

Since the Agent Data Platform launch (early November 2025), memory is platform-native. Three layers: per-turn save, conversation-start boot, agent-decided implicit save. [0:58:20] [1:01:32]

5. Outcome pricing = incentive alignment

Align with customer value (sale made, retention saved) rather than tokens used. Sierra mostly doesn't get pedantic about which exact token counts for what — alignment is more important than precision at early stage. [1:15:00] [1:18:23]

Where Zack disagrees with the broader AI-agent discourse

Three places Zack pushes back explicitly:

  • Multi-agent = org-chart shipping. Sub-agents without truly separable contexts destroy value; one monolith with great context engineering usually wins.
  • "Whenever you think the model's too dumb, the model's actually too smart." The lever is context, not model.
  • "No breakout memory company yet." Memory is essential but trust-bundled with auth. B2B buyers worry about identity in a way consumer products don't.

🧱 Monolith vs Multi-Agent — Zack's central argument

Zack is explicit, on-record, and repeated (this is the central claim of the episode): multi-agent systems are often not useful, and the patterns people reach for them are usually "shipping your org chart." Sierra's brand-representing agent is typically one agent; decomposition happens only when context is truly separable.

"If you want a multi-agent system so that one team can work on one agent and one team can work on another agent, then you're shipping your org chart." [0:46:21]
"By building it as a multi-agent system, you're often depriving of the agent doing the task of the information from the triage and depriving the agent doing the triage of all of the procedural information from the task. And that's typically destructive of value." [0:47:00]
"I'm kind of a monolith loyalist on that." [0:48:29]

When Sierra does use multi-agent

Zack isn't anti- multi-agent. He's pro- deliberate-context multi-agent:

  • Truly separable jobs. "Where there's not any purpose of the first context being part of the second context." [0:48:09]
  • Long-running sub-agents in the background. Sierra agents can call out to other Sierra agents (or in-house platforms) for deep research, document generation, etc. "You might have a long-running agent and then the Sierra agent can call out to it and wait for [the result]." [0:18:38]
  • Expert offload. Customers with document-generation experts in-house have the Sierra agent call them; the front agent handles brand voice, the back agent handles specialized output. [0:18:50]
  • The "kicking off even a sub-agent" test. "If you're kicking off even a sub-agent, you want to make sure that it has everything it needs to do that task and that there's no reason why it shouldn't just be part of the main agent." [0:47:36]

Sub-agent mechanics inside a Sierra voice conversation

Sierra supports both directions of agent-to-agent call, and the dominant protocol isn't A2A — it's API calls ("you save a lot of tokens and make sure that you're 100% accurate that way"). [0:19:47]

"Sierra agents support the MCP and agent-to-agent protocols. You can kind of install that integration and then your agent can be an MCP client. You can also set up your agent to be an MCP server. So this is how we support ChatGPT apps." [0:19:47]

Why context engineering usually beats decomposition

"If you're managing context correctly, if you're doing really really good context engineering, then typically it's just not a problem because you're not exposing the wrong context to the wrong agent." [0:48:04]

Caveat — Zack's honest disclaimer

"In my personal opinion with you know, May 18th, 2026, there not a lot of great times for it [multi-agent]." [0:48:09]

🗣️ Voice architecture — modular, parallel, multilingual

"Voice has been maybe the most fun project that I've worked on in my whole career." [0:51:15] Sierra's voice stack is structured around three architectural decisions that compound — and that you can break apart individually:

1. Parallelize thinking, listening, talking

The single biggest unlock:

"One of the big unlocks for Sierra agents was deciding to think about not only how to parallelize a task, but how to parallelize thinking, listening, and talking. So that when I'm listening, I'm already thinking about what I might say next. When I'm talking, I'm listening for interruptions." [0:52:56]

Compare with the simplistic stack:

Old / naive

"~50 lines of Python (Silero VAD) deciding when to speak + a trillion parameters deciding what to say. That balance feels very off." [0:51:48]

Sierra's answer

Three concurrent streams: transcription + reasoning + synthesis. Each can use a different model; each runs in parallel; each can be swapped per region / language / customer. [0:52:56]

2. Modular, multi-home providers

"An early decision that totally played out to our advantage — multi-home providers across transcription, synthesis, and native voice-to-voice models." [0:27:23]

Why: no single provider is best across all dimensions. The "best" transcription provider for English may hallucinate during silence; another provider wins on quality for a specific language. Sierra ensembles them.

3. Ensemble transcription for hard languages

"We have, I think, almost about 60 languages on Sierra platform. ... Sometimes the very best transcription provider might have a 20% word error rate. I think that's true for a language like Hungarian. And so ... how can we ensemble multiple transcription providers in order to get that down and kind of be better than a single model is on its own." [0:50:25]

Voice-to-voice models (live in production)

"As of a few months ago, we now have production agents live with the voice-to-voice models. ... You still need the transcript in order to make API calls and that sort of thing, but the agent is responding with audio as the input." [0:50:05]

Notable mention: ChatGPT real-time — "real-time I think they might be up to 2.5 at this point. They've had like three big real-time launches at OpenAI this year already." [0:50:25]

How Sierra hydrates voice-to-voice as "last-mile"

"Basically using the entire Sierra pipeline and then holding on to the input audio and piping that in with all of the prompt context into the audio model to do the last mile." [0:54:34]

Sierra's posture: use the closed audio model for last-mile audio generation only, but keep the orchestration / context / tools routing in their own pipeline. They considered going all-in on the closed model and chose not to — "you can either do the same thing a little bit more easily or you can set your sights on new and more impressive things." [0:54:50]

Why voice has more friction than chat

Three friction points raised by Zack: (a) latency (1-2 s budget), (b) multilingual fluency (~60 languages, with WER variance), (c) interruption handling + VAD (when to decide not to speak is non-trivial). [0:49:51]

🧠 Context Engineering + Memory — first-class primitives

Context engineering — the definition

"Showing agents everything they need to do the right thing, but nothing more. And as models get smarter, you can be a little bit less precise with everything they need, and certainly less precise with nothing more. So, early on the agent SDK was really about only giving the relevant information into the model. Now, that's become a system reminder, or you can just kind of ask it." [0:40:01]

The 80/20 rule on abstractions

"My personal perspective is that 80% of the time you want to do the first thing and just meet the models where they are on their turf and you should reserve the second one for that really special case." [0:11:52]

Where "the first thing" = use what models are trained on (file systems, git, grep, LangGraph [0:12:53]); "the second thing" = custom DSLs / formats not in training data.

Memory as a first-class platform primitive

"Memory is very important to the platform. ... since the agent data platform introduction, which we launched back in early November, it's been a first-class primitive on Sierra." [0:58:20] [0:59:01]

The three layers of memory

Layer 1 — per-turn save

"On a given conversation turn, you could say, 'I want to save this to memory.'" [1:01:32]

Layer 2 — conversation-start

"At the beginning of a conversation, you could say, 'These are the things that are important to remember.' ... 'Remember their birthday.'" [1:01:32]

Layer 3 — implicit agent-decided

"If the customer said, 'Hey, I want you to remember this when I call back in the future.'" — or let the agent decide. [1:01:52]

Why memory is not a breakout company yet — Zack's bet

"I feel like memory is this really hot topic, and everyone loves to talk about it. ... but I don't see any of them being massive breakout successful. Why is that?" [0:61:17] (Harrison asks)
"When you're trusting us with memory, you're trusting us with authentication. ... in order to pull off memory, you need to be trusted with something that has higher risk. ... I would imagine you're biting off more than you think when you sell memory." [1:03:00] (Zack answers)

Sierra's take: memory startups under-estimated how much identity trust is bundled with the surface they want to use. Consumer products (ChatGPT, Claude) have that trust "by default"; B2B memory startups don't. Memory as a category is real; memory-as-a-standalone-B2B-product is unproven.

Memory structure — what Sierra uses internally

"When you think about the structure of memory itself, do you guys think about it as a knowledge graph, a vector store, a file system?" [0:61:45] (Harrison)

Zack dodges the schema question — and arguably rightly: the storage substrate is less important than the agent-side policy on retrieval. Translation: "we have one, but the schema question is the wrong level — ask me about the policy." Worth re-listening for: [0:61:45]

What memory buys you — measured outcomes

"If you know who someone is, you greet them by name, you remember what's important to them, and you show empathy in the moment, it increases all of the metrics that are most important to businesses, from resolution rate to conversion rate." [0:58:48]

Continual learning — auto-detect → Ghostwriter → review

"Where we are today is you can automatically detect an issue with a monitor. Ghostwriter can automatically suggest a fix to an issue. And you can review that issue and push it to your agent. And so you're still in the loop. ... In the near future, you will start to see the first cases of Sierra agents improving themselves where they have a confidence level to the fix." [1:06:17]

Self-improvement primitives exist today; the gate is confidence threshold, not capability. "It's just around the confidence that people have in the level of control that they want to have." [1:07:09]

⚙️ Ghostwriter + Agent Data Platform — Sierra's internal dev / observability stack

Ghostwriter — "the Codex for building agents"

"Ghostwriter is the agent similar to Codex or Claude code for building agents." [0:04:06]

What Ghostwriter writes

  • Not code. Ghostwriter writes Journeys directly — so the customer can inspect and edit what was generated. [0:10:45]
  • Domain-aware. "Ghostwriter just kind of already knows those concepts and is an expert in journeys." Knows "flight booking," "car rental," "referral from primary care provider to a specialist" natively. [0:10:45]
  • Learning curve obliterated. Customers come in and describe a workflow in plain language; Ghostwriter drafts the journey. [0:10:20]
  • Proactive suggestions. From analyze, Ghostwriter proactively suggests an improvement based on an issue spotted. [0:06:32]

Agent SDK + Journeys — the isomorphic compilation

"The no-code journeys layer compiles down to Agent SDK code deterministically and isomorphically ... you can turn it one way and then turn it back and it's the same. ... you can have code that you transition over to no code, you can have no code that you transition over to code." [0:09:14]

This isomorphic property is unusually strong — most no-code tools have a "round trip" that loses information. Sierra's choice means: an engineer can pull a journey into code when determinism matters (e.g., CI/CD integration) without losing fidelity. [0:15:00]

Agent Data Platform (ADP) — launched Nov 2025

"Agent Data Platform ... brings together ML data / big data about customers, marries that with in-the-moment context. ... Can either integrate with your CDP, all your internal systems, or you can have it just on Sierra, or a zero-copy integration." [0:33:16]

ADP capabilities

🛒 Real example: sales

"During sales, oftentimes there's structured data that knows the right offer to present, but doing it just with that structured data wouldn't necessarily work. Pick different offers based on conversation context and pick the right one for the moment." [0:33:43]

ADP + ML — the data flywheel

"We do have our agent data platform and there are machine learning models on top of it. But they don't necessarily know what you care about at a level deeper than that. Zack's interesting admission: classical recommendation systems can outperform new LLM-only approaches for some knowledge questions. The Sierra position is "use both" — the LLM for context-and-conversation, the ML for the "what does this person actually want to be offered right now" question.

Why no fine-tuning on regurgitation-prone data

💰 Outcome-Based Pricing — mechanics, alignment, when it works (and doesn't)

Sierra is the canonical example of outcome-based pricing in production at scale (already on commission for sales outcomes). Zack walks through what this means, why most products can't do it yet, and the open questions.

What "outcome-based" actually means at Sierra

"There are cases where the Sierra agent is actually, because of our outcome-based pricing model, getting paid a commission on a sale ... cases where we differentiate [pricing] within those different types of requests." [0:02:08] [1:18:23]
  • Sales / membership: pricing is a function of the resulting commercial value.
  • TV signal reset / bank balance check: low-complexity, lower-impact — pricing is smaller.
  • Complex device troubleshooting: 15-turn conversations, large context-engineering cost — pricing reflects that.
  • Question-answering: "not an area where you would have a high premium for an outcome of any particular sort." [1:15:35]

The Sierra pragmatic stance on granularity

"We're not dogmatic about it. What we found is that often times the benefits of having our incentives aligned are so high that it's not worth negotiating every detail of what counts for what and it kind of makes sure that incentives are deeply aligned. And then on top of that I think you can get really pedantic about the engineering of specific outcomes and maybe over time the market will move in that direction. But I think you're missing the forest for the trees if you do that today." [1:18:23]

Why outcome pricing isn't everywhere yet

"I think it's probably a bit of both. ... A lot of products find themselves in a position of being more similar to what you could just buy today." [1:19:07] Zack's ranking of reasons:

  1. Commoditization pressure — many AI products approximate a thing you can already buy (knowledge lookups, FAQ answering). When the work is commoditized, usage-based or seat-based pricing dominates. [1:15:35]
  2. Operational complexity — "the second one is more important" — the operational lift of proving an outcome happened is non-trivial for low-value flows.

Where outcome pricing is heading

"Where we're seeing things going is intra-conversation outcomes, to also thinking about more — as I mentioned — kind of the moments that matter across the customer life cycle." [1:15:42]

Zack's forward claim: outcome pricing will migrate inward — from "did a sale happen" (outer) to "did this conversation successfully progress" (inner). The further in, the higher the engineering overhead.

Why Sierra can sustain outcome pricing (vs e.g. Claw that can't)

Harrison explains that because Sierra charges per-tenant and the customer can stand up a new contract per workflow, Sierra can take on the "outcome" risk that B2B SaaS without that structure can't. [1:15:35]

Open questions (Zack's own)

Zack doesn't claim outcome pricing is solved — and acknowledges the unknown unknowns:

  • "Maybe over time the market will move in that direction" — implying a long arc, not a sudden flip.
  • "You're biting off more than you think when you sell [outcome-priced] memory." [1:03:00] — applies to many B2B risk surfaces.

🤝 A2A / MCP / API — which wins, the 80/20 rule, the actual Sierra stance

One of the most quotable exchanges in the interview. Zack is direct: the most common agent-to-agent call at Sierra is a plain REST API. MCP and A2A exist, but they solve different problems.

"The most common is an API call. When you know who you're talking to in advance, often times you can save a lot of tokens and make sure that you're 100% accurate that way." [0:19:47]

Why API > MCP / A2A when caller & callee are known

Three reasons Zack mentions (implied, one explicit):

  1. Token savings — known signature, no schema discovery.
  2. Determinism / accuracy — "100% accurate that way" vs probabilistic schema matches.
  3. Latency — direct call avoids negotiation overhead.

When MCP wins

"Sierra agents support the MCP and agent-to-agent protocols. You can kind of install that integration and then your agent can be an MCP client. You can also set up your agent to be an MCP server. So this is how we support ChatGPT apps which rely on MCP servers." [0:19:47]

MCP wins when:

  • You don't know who you'll be talking to at runtime.
  • You need a common schema across many callers / servers.
  • You're a "server side" of a ChatGPT app (Redfin example).

When A2A wins

A2A is mentioned as "supported" but not the most common. The implicit Sierra position: A2A is for true cross-org agent calls where neither side wants full coupling and discovery is the value.

The cross-reference interaction

"[MCP means] the tools of the agent can be made available to ChatGPT and then you can at-reference a Sierra agent. The example that would be most familiar is Redfin." [0:20:16]

Translation: a Sierra agent at Redfin exposes its tools via an MCP server. ChatGPT receives those tools when the user references the Redfin agent, and ChatGPT executes those tools through the Sierra agent. End user talks to ChatGPT; ChatGPT talks to Redfin's Sierra agent. [0:20:16]

The 80/20 thesis on technology choice (model-agnostic)

Zack's meta-rule applies beyond Sierra: 80% of the time, use whatever models are best at. Tools, protocols, frameworks — same pattern. "At least Anthropic models recommend LangGraph for a lot of use cases and that's great. ... people choose technology because the models are really good at writing it." [0:12:53] [0:11:52]

⚠️ 反方 / 局限 — blind spots, pushbacks, what Zack's argument doesn't address

Credibility is medium (industry analysis from a Head of Product at a company with skin in the game — Sierra's product, Sierra's pricing, Sierra's eval suite). Each item below is a Sierra / LangChain-side claim that has known counter-positions; treat them as engineering hypothesis, not gospel.

1. Monolith loyalism may be Sierra-specific ⚠️

Zack's claim "multi-agent often not useful" comes from a customer-agent workload where one brand = one agent. In a research / coding / data-pipeline workload, multi-agent is the rule not the exception. The monolith-stance is correct only when context quality is high; in low-context environments multi-agent may still win. [0:46:21]

2. "Model is too smart" — sometimes the model actually is too dumb ⚠️

The quote is rhetorically sharp but not always true. For novel multi-step tasks with no training-data precedent, model quality is the bottleneck. Apply the 80/20 — context fixes 80% — but don't over-apply. [0:41:43]

3. "No breakout memory company" could be a market-temporality call ⚠️

Zack attributes this to identity-trust bundling — but the standalone-memory startup failure rate could also be because the layer is naturally bundled into platforms (ChatGPT, Claude, Sierra ADP). A "breakout memory company" might emerge once memory becomes a portable primitive (similar to how Stripe unbundled payments from merchant sites).

4. PCI Level 1 ≠ voice-payments are solved ⚠️

Sierra is the only launch-time voice-payments-without-transfer platform. That's an internal claim. Whether the underlying UX (co-browse, dispute, refund, post-chargeback reconciliation) is actually solved is open; the architecture enables it but doesn't prove it.

5. 60 languages ≠ fluent 60 languages ⚠️

"Almost 60 languages" with a 20% word error rate on Hungarian for the best single provider (and ensembles needed) is a real engineering achievement, but "fluent" can mean many things. Production-fluency numbers per language aren't given. [0:50:25]

6. Outcome-based pricing suppresses competition ⚠️

Outcome alignment is great for high-margin businesses. But it can shut out small customers where outcomes are hard to measure, and it creates audit overhead. "I'm not dogmatic about it" — but the longer-term test is whether outcome-pricing scales beyond Fortune 100. [1:18:23]

7. "Agent-to-agent via API" hides A2A's true value ⚠️

Zack's "API calls win when known in advance" is true — but the trend Zack doesn't push on is cross-org calls where the callee is unknown. A2A's value grows as more agents from different vendors have to discover each other. Sierra's posture is "fine, we'll add A2A when we need it" — fair, but not advocating for the protocol.

8. The voice-to-voice "last-mile" pattern is fragile ⚠️

"Holding on to the input audio and piping that in with all of the prompt context into the audio model" works when (a) the audio model accepts long context, (b) Sierra's pipeline doesn't drift from the audio model's training, (c) latency holds at 1-2s. Any of those three breaking — particularly closed-vendor API changes — is a Sierra-side risk. [0:54:34]

9. "Tau cinematic universe" — sponsored eval ⚠️

Tau-bench is a vendor-published eval that has outsized influence on the field. Sierra has a stake in defining "what good looks like." Acceptable, but treat it as Sierra's view of good, not an independent benchmark — and re-run on your own data. [0:42:10]

10. "Isomorphic round-trip" is a strong claim — under-pressure? ⚠️

"You can turn it one way and then turn it back and it's the same" is a strong guarantee. Most round-trip tools lose fidelity at edge cases. Sierra claims this holds for Journeys ↔ Agent SDK; hasn't been independently verified at production scale. [0:09:14]
