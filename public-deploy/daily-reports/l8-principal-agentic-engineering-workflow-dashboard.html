① Why Terminal > GUI? = 2 大理由

2 大理由

  1. "Hands never leave the keyboard" = 保持 flow
    • "When your hands stay on the keyboard, you stay in the flow"
    • "If you have to move your hand to the mouse every couple of seconds, it breaks the flow and forces your brain to contact switch"
  2. Same workflow everywhere = 跨设备一致性
    • "I can keep the exact same workflow everywhere, even on my phone"
    • GUI apps = paradigm 不一致 ("hard to build the discipline of hands on keyboard")
"Terminal apps, on the other hand, are all designed for the keyboard, so there is no reason for your hands to move anywhere else."

② Ghostty + tmux + Neovim

Ghostty (Terminal Emulator)

26K GitHub stars

Built by a guy named West

Truly cross-platform

"the only terminal emulator I can find that can work on windows exactly the same way it works on Mac and Linux"

Highly customizable

Lua scripts (westerm.lua) + hot reload

tmux (Terminal Multiplexer)

#能力实战
1 Split panes agent + editor + shell
2 Tabs/windows 多 session 并行
3 Persistent sessions detach 后 state 保留
4 Cross-device connect "I can connect to this same session from another device, like my laptop or my phone"
"That's a real game changer. That's very hard to replicate without this terminal-centric workflow."

Neovim (Text Editor)

  • Modern version of vim + keyboard-first
  • Plugin ecosystem = <leader>S search · <leader>F find files
  • Relative line numbers = 跳转更快
"Whenever I'm in vim, I'm just flying like a bird and it's awesome."

③ 4 大 Agent Harness 评测

Harness优点缺点Kun 评价
Claude Code Most sensible default out of the box · rich feature set Sometimes buggy · not very customizable Anthropic 订阅唯一实用选择
Codex CLI Smoother (Rust) · open source · 自检自身 source Lacking bells and whistles · not customizable "Codex can inspect its own source code"
Pi Coding Agent Minimal · highly extensible · tinkering 无 bells and whistles "philosophy is to be minimal and highly extensible"
OpenCode Battery smooth TUI · good model integration · complete out of the box - "model agnostic · grab from shelf and go"

Kun 选 Claude Code 主线

"I'm going to use Claude Code because I know many people are already familiar with it, but I have been very strict about making my workflow agent agnostic because the landscape is changing very, very fast."

Agent-Agnostic 哲学

"Who knows which model or agents will be the best performing one next month? So everything I show here in the video is agent agnostic and should be applicable regardless of which model or harness you use."

④ Onboarding 1: Memory Files

Global Memory File

  • Location: ~/.claude/CLAUDE.md (Claude Code)
  • 极简 = "because everything in this file gets loaded into the system prompt of every single agent session across all our projects"

Examples (Kun 实际写的)

Personal Preferences

"Never use em dash"

"Somehow AI models are trained to use em by default instead of a plain dash, so now whenever I see em, I just feel like it's robotic."

⭐ Technical Decisions

"When making technical decisions, don't give too much weight to development cost"

⭐ Bug Fixes

"When doing bug fixes always start with reproducing the bug in an end-to-end setting"

Project Memory File

  • Location: <project>/.claude/CLAUDE.md
  • 比 global 详细多 = "basically the collective learning of all the agent sessions in this project"
  • 包含：project context + repo layout + terminology + key components + E2E testing + conventions

Symlink 技巧

"I use this command ln -s to make CLAUDE.md a symbolic link to AGENTS.md. So they both exist, but under the hood they point to the same file."

记忆积累哲学

"The way I built this file is not by writing everything by hand, but rather that every time I saw the agent doing something wrong, I would correct it and ask it to remember to not make the same mistake again by storing the learning into this memory file."

⑤ Onboarding 2: Skills + Skill Creator

Skills = Conditional Knowledge

  • 不是每条 knowledge 都要 in memory
  • 部分 conditional 知识 = "only needed if the agent is making changes" → 移到 skills
  • 流程：跟 agent 说"let's extract the E2E testing instructions into a project level skill"

Skill Creator

Anthropic 官方 skill for "教 agent 怎么创建新 skill"

安装：npx skills add @anthropic/skills

支持：any agent

⭐ Skills = Progressive Disclosure

"Skills is designed for progressive disclosure, which means when your agent starts, it only loads this tiny description field from your skills into the system prompt to know what these skills do, and only when it actually decides that it needs to use a certain skill, it will then read the rest of this file."

Token 节省

"This allows you to store a lot of the knowledge about how to do various kinds of things without blowing up your system prompt and memory file with a ton of contents that uses your tokens for every single request, whether the request actually needs those skills or not."

⑥ Voice Input = 3x Faster than Typing

Kun 实战

  • 完全移向 voice input = 几乎不再 typing
  • 工具：Open Super Whisper（完全 free + open source）
  • 本地跑 Whisper 模型 = "the quality is like really, really good"
  • 例：直接说 "explain this repo in a concise way"

Stanford Paper

"Talking is three times faster than typing. So this is a very big boost in productivity."

Fall Back to Typing 唯一场景

"The only case where I fall back to typing is when I need to give the agent a URL or a file path, or something like that. Trust me, you don't want to speak a URL out loud, whether it's by yourself or with other humans around."

Irony

"What is the CEO of Anthropic doing here? Apparently Dario was doing some speech recognition stuff back in 2016. Now we're using speech recognition technology to talk to Claude, which is also created by Dario."

⭐ 关键 Insight 1: AI Self-Underestimate

Kun 实测 (3D FPS Game)

"If we ask a frontier model to estimate the development cost of a project, let's say I want to build a 3D first person shooting game that I can play locally with AI enemies, how long do you think that will take?"

- Model 估: days / weeks / months

- 实际 AI agent 跑: just a few minutes (Kun 多次实测)

根因

"This mismatch is happening because the models were trained from human data, and that is what a typical human developer would give as the estimate. AI doesn't seem to know it can code much faster than humans yet."

Bias 结果

"When AI is making technical decisions, it's implicitly assuming the development cost for some of the options are much higher than they actually are. This biases the model to choose cheap solutions that are often low quality, not scalable, or hard to maintain."

修法

~/.claude/CLAUDE.md 写：

"When making technical decisions, don't give too much weight to development cost"

⭐ 关键 Insight 2: E2E Testing > Unit Testing

AI 默认写 Unit Test 问题

"AI models today by default like to write unit tests, which are often not sufficient and not really covering the product behaviors we want to guard."

Kun 哲学

"When doing bug fixes, always start with reproducing the bug in an end-to-end setting as closely aligned with how an end user would experience it as possible."
"I found that leaning into end-to-end testing is a lot more reliable."

实操流程

  1. User reports bug
  2. Reproduce E2E first（像 end user）
  3. Fix
  4. E2E test passes
  5. (Optionally unit test for the specific fix)

⭐ 关键 Insight 3: 慎装 Internet Skills

2 大风险

🔴 Security 风险

  • "These skills can instruct your agents to run pretty much anything on your machine."
  • "The agent can leak your API keys or even credentials to your bank account to untrusted third parties without you knowing."

🟡 Performance 风险

  • "Some of the skills actually degrade your agents performance."
  • "Some of the skills actually degrade your agents performance."

Kun 实测反例

"Anthropic Skills" repo (177K GitHub stars)

"evaluated with Program Bench, which tests the agent's ability to build programs end to end"

结果：5% more tokens + worse results

"And if you look closely, this skill is not even written by Andrej Karpathy himself."

Kun Rule of Thumb

"Do not install any skill from the internet that claims to magically make your agent perform better, but hasn't published anything rigorous that proves its claim."

Stars ≠ Quality

"A lot of the skills being widely shared today have not been rigorously evaluated, and are typically just some random guy who found something that worked for themselves and somehow got it to go viral. Their GitHub stars only tell you how popular they are and not whether they are actually helpful."
