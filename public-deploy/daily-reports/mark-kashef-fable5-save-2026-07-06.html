三种打法的对比 (binary choice → third move)

Fable 5 真正不可替代的是什么?

答案:不是智力(intelligence),而是 elicit Unknown Unknowns 的能力。

便宜模型(Opus 4.8 / GPT 5.5 / GLM)智力够,但缺少 Fable 5 那种"把你领到你从没想过的领域"的能力。Wargame = 把这个能力外化(externalize)成 markdown blueprint,让便宜模型拿着这份 brief 也能跑到 Fable 5 同等水平。

Known/Unknown 4 象限 - orchestrator 盲区框架

"with a model this capable, you're no longer held back by raw intelligence. You're held back by the things that you don't know as the orchestrator, as the person providing the instructions, the plans, and the prompts to these kinds of models."
- [01:51] 引用 Anthropic 工程师文章
✅ 我们知道我们知道的 (Known Knowns)

项目启动时已经掌握的事实:技术栈、需求、约束。模型可以理解,人也知道。

⚠️ 我们知道我们不知道的 (Known Unknowns)

意识到存在但不掌握的盲区:性能基准、安全漏洞、边界 case。模型可以列出清单,需要人去补。

🟡 我们不知道我们知道的 (Unknown Knowns)

Tacit knowledge - 你以为模型知道的生活经验、直觉、行业常识,但即使 Fable 5 也没有你那个 level。需要显式化。

🎯 我们不知道我们不知道的 (Unknown Unknowns)

Fable 5 真正不可替代 - 你不知道这片领域值得探索,但 Fable 5 能领你到这里。便宜模型够聪明,但不会主动探到这里。

📌 Wargame 的核心任务:

在 Fable 5 还在的时候,用它的智力 + 你不知道的盲区,把所有可能的"uncomfortable question"问出来,固化成 brief。然后把这 brief 喂给 Opus 4.8 / GPT 5.5 / GLM - 它们拿着这份地图就不会迷路。

🎯 War-gaming 定义:不是 Plan,是 Plan 的极限

Wargame vs Plan - 5 维对比

维度Plan (传统)Wargame (Mark 提出)
假设 线性成功路径(blue-sky scenario) 枚举所有失败分支(pessimistic + optimistic)
结构 Phase 1 → 2 → 3,按时间线 Move → Reaction → Counteraction,按回合
每个 step 内容 任务 + 预期 动作 + observation + failure signal + failure cause + countermove
执行者 写 plan 的同一个模型(昂贵) 便宜的 executor 模型(Opus 4.8 / GPT 5.5 / GLM)
失败成本 80% 烂尾,20% 最难的留给你 预先枚举好应对路径,执行时遇到 → 查 brief → 走 counteraction

三大元素(每个 move 必须包含)

1. Action - 动作

这一步 AI 要做什么具体动作(e.g. 调用 API、读文件、修改 config)。

2. Reaction - 现实反应

执行动作后,现实"humility it by throwing some form of error" - 哪个具体的 error、长什么样、exit code 多少。

3. Counteraction - 应对

检测到 error 后的应对路径 - 重试 / 回滚 / 改方案 / 升级到 human-in-loop。

具体例子:website API endpoint build

Move 1: AI 调用 POST /api/endpoint {param1, param2}
  Observation (success): HTTP 200 + {result: {id: "xxx", status: "created"}}
  Failure signal: HTTP 500 + {error: "DB connection timeout"}
  Failure cause: 可能是 connection pool exhausted / DB restart 中
  Counteraction:
    - 先 retry 1 次(SET retries=3 in client)
    - 仍 fail → 切换到 read replica
    - 还 fail → flag 到 ledger.md 加 <db_failures_today> placeholder,等人工查

Move 2: 验证结果数据完整性
  Observation (success): SELECT COUNT(*) 跟写入数匹配
  Failure signal: COUNT 不匹配(这是 silent corruption)
  Failure cause: race condition / partial commit
  Counteraction: 触发 reconciliation job + 通知 Slack #incident

Move 3: 返回 response 给前端
  ...
🧠 为什么这是 plan 的极限?

普通 plan 写"调用 API"就完了。Wargame 把每一步的"如果失败怎么办"全部显式化 - 这就是 Second/Third/Fourth order consequence 推演。执行者拿到这份 brief 即使智力不如 Fable 5,也不会"卡在 20% 最难的部分"。

📋 Prompt 模板 - 7 段固定结构

完整模板(每个 use case 都用同一段头)

War-game order.
You are not executing this mission, you are purely war-gaming it.
A cheaper executor model will run the brief below. [指定 executor: Opus 4.8 / GPT 5.5 / GLM / Sonnet 5]
[可选] Tailor your prompt and war-game to how [executor] would execute it best.
参考 [executor] 的官方文档 / system card / API reference。

Then fight the mission on paper move by move and write it to
war-games/<name>.markdown

每个 move 必须包含:
- Expected observation: 成功看到什么
- Failure signal: 失败看到什么(包括具体 error message / exit code)
- Failure cause: 哪类问题(常见:timeout / permission / data race / validation)
- Countermove: 应对路径(retry / fallback / escalate)

每条分叉(像迷宫分叉)给 trigger:
"If you observe X → take route Y, otherwise route Z"

最后收尾 3 段:
1. Assumptions flag - 侦察 / due diligence 解决不了的变量,显式 flag 给用户
2. Abort conditions - 什么情况整个 plan 停(无权限 / 资源耗尽 / 安全违规)
3. End state + verification - open every page / exercise every link / 跑 smoke test

[Mission brief 部分:use case 特定]

# Website example:
I'm rebuilding the marketing site for [business name].
Current site: [problem description].
Target audience (ICP): [who].
Call to action: [what you want them to do].
Framework: 你的选择。
Sections: [list]。
URL: [link]。
Mobile-first 是硬要求。
Verify before reporting: open each page, exercise every link.

关键设计原则

原则为什么
"You are not executing" 写在最前 Fable 5 容易默认"既然你给我 brief,我就执行"。这句话锁死它只推演,不实际跑。
显式指定 executor 模型 Fable 5 可以按目标模型 docs 优化 brief(例如 Opus 4.8 vs Sonnet 4.6 文风不同)。
"move by move" 防止 Fable 5 给你一份 high-level plan 而非细粒度 brief。
write to war-games/<name>.markdown 文件输出强制 - 后面 /goal 命令需要批量读取。
每个 move 4 元素 + abort 这 4 个元素 + abort condition 是 wargame vs plan 的本质区别。少一个就退化回 plan。
💡 进阶:Claude Code Guide sub-agent

把 wargame 文件喂给 Claude Code Guide agent + prompt "tailor this to how Sonnet 5 would execute" → sub-agent 自动读 Sonnet 5 system card + 文档 → 输出目标模型专属版 wargame。精度再上一层,但需要 executor 模型已发布。

每个文件的角色

文件写什么谁来写
tasks/<name>.md Mission brief(用 5 个 use case 模板填空) 你(orchestrator)

/goal + /loop 双命令 - 2 层 fan-out

/goal - 一次性 fan-out

Prompt: "Every mission file in /tasks has a first draft war game. Go through each one, loop through, and do the recon mission for each mission, write a war games file, and then draft all 10 before polishing any."

效果:Fable 5 fan-out 10 个 sub-agent 并行写 war-games/* - "a million tokens later, you could have a war games folder that's fully populated"。

关键:draft all 10 before polishing any - 先全 draft,后面 /loop 再 polish,避免一头扎进去单个 polish 浪费 token。

/loop - 周期性迭代

Prompt: "theoretically run /loop every 20 minutes, loop through every first draft until you really push it to its limits."

效果:每 20 分钟 Fable 5 对每个 wargame draft 重新跑一轮 - 看能不能再加 countermove / assumption flag / 更细粒度的 failure signal。

警告:⚠️ Mark 没解释为什么 20 分钟 - 推测是 token cost 平衡,但没数据支撑(见 Tier 4 §video gaps)。

ledger.md 协议详解

"the whole point of it is pointing out anywhere where it's blocked. So, if it's war gaming and there's some form of variable that's undefined, it should populate a little parentheses variable placeholder for something that it needs your input"
- [11:06] Mark Kashef
