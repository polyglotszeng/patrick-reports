Test 1 — Agentic 3D 平台

同一 prompt · Fable 5 走 Claude Code · Soul 走 Codex · 完整 3D 跑酷游戏

🥇 Fable 5 (左侧)

Bird's eye 俯视角

  • 时间: 21m 37s
  • 成本: $14.22
  • 输出 token: ~90,000
  • 风格: top-down, 城市地图, 收集金币, 跳跃 + bunny hop
  • Nate 评价: "functionality-wise, the coins are working, the controls work. I just jumped over a building and I can bunny hop."

🥈 Soul (右侧)

POV 第一人称

  • 时间: 23m
  • 成本: $4.5
  • 输出 token: ~31,000
  • 风格: POV, 3D 城市, 跳跃 / 旋转 / 物理 ramp
  • Nate 评价: "pretty similar controls" — 都跑通了, 但视角选 POV 而非 bird's eye

📊 对比解读

  • 成本差 ~3.16× (Fable $14.22 vs Soul $4.5)
  • Token 差 ~2.9× (90k vs 31k)
  • 时间差 ~1.4m (Soul 反而慢 1.4m, 罕见)
  • 质量: Nate 明确说 "Fable's output was that 20 times better" — 否认. "I would have taken Fable's output in this one as well."
  • 解读: Soul 在 agentic 多步推理下不一定比 Fable 快, 但 token 节省显著

⚠️ Token 效率 ≠ 质量 — 关键洞见

"I don't necessarily think that it was [20× better] but just because we're going off of like the actual quality I'm going to say that I would have taken Fable's output in this one as well. All right so I did one more side-by-side experiment."

Nate 多次强调: token 少 ≠ 输出更好. Soul 输出 31k tokens 跑通了, Fable 输出 90k tokens 也跑通了. Soul 节省 token 是 harness + 训练方式的结果, 不是"模型更聪明".

📐 "Unit Economics" 模型

Nate 提出一个决策框架:

"Let's say the task is a five. Do you need a model that is a 10? Probably not. You only need a model that's maybe like a five or a six. So in this case with these quick one-shot, you know, API requests probably stateless, Soul seems to crush Fable here."

含义: 如果你的任务是 5 分难度 (中等 one-shot), 用 Fable 10 分能力是浪费. Soul 5.6 能力 ≈ 5-6 分 = 完美匹配. 这跟 Anthropic Fable tier 定价逻辑吻合 — Fable 贵是因为"做 10 分任务", 不是因为"做 5 分任务更慢".

🤔 Fable "拒绝回答" 现象

Fable 多次直接 refused 不答. Nate 评价:

"the security guard rails are baked in pretty hard"

Anthropic 把 Fable 安全护栏做得非常 hard — 这是为什么 Fable 在某些 query 上不答,而 Soul 会答. 这解释了 24:3 胜负比里 Fable 的"低胜率" — 不仅是质量, 也是 意愿. 含义: 如果你需要模型 100% 回答 (即使有边界问题), Soul 比 Fable 更可靠.

🧮 关键洞见: 即使价格相同 Soul 仍更便宜

"Let's say that Fable and Soul costed the exact same from an input and output token perspective, Soul would have still been meaningfully cheaper because it just was more efficient with tokens. I don't know exactly why that is, but it just seems to be so much more efficient with tokens.   2. Computer use 任务优先 Soul — Nate 明确 Soul 在 computer use 维度更强. 浏览器自动化 / GUI 操作换 Soul.
  3. 评估自己任务难度分布 — 5 分任务 vs 10 分任务占比. 如果 90% 是 5 分, Soul 完全够用.

🛠️ T2 — 短期 (2 周内)

  1. 设计 Fable-orchestrates-Soul 架构 — Nate 推荐的最佳实践. Fable 做 manager (判断 / 拆解 / 派活), Soul 做 worker (执行 / 测试 / 验证). 两层 cost model 拿 Fable 质量 + Soul 速度.
  2. 监控 Soul over-engineer 风险 — Soul 跑太多测试是常见 pattern. 加 cost ceiling 或 token budget 触发 early stop.
  3. Solon Ultra 谨慎使用 — 容易 over-think. 默认用 Solon (非 Ultra), 仅在确实需要时切 Ultra.

🔮 T3 — 中期 (1-3 月)

  1. 追踪 GBT 6 发布信号 — 如果 Nate 预测对, GBT 6 才是真正对标 Fable 的下一代. 关注 2026 Q3-Q4 模型发布.
  2. 设计 latency budget — Soul P50 快 P99 慢的分布特点. 生产环境用 Fable 跑长任务 (稳定), Soul 跑短任务 (快). 不要混用.
  3. 评测自己的 Fable vs Soul one-shot benchmark — 拿自己的 24 个 one-shot 任务实测, 验证 Nate 24:3 比例是否在自己场景成立.

🧪 T4 — 长期 / 探索 (3+ 月)

  1. Fable + Soul 多 agent 编排框架 — 不是 Claude Code 内部调用, 而是独立 orchestration layer. 类似 OpenClaw / Hermes 的"manager + worker" 模式.
  2. 模型拒绝率统计 — Fable "guardrails baked in pretty hard" 是已知问题. 跑 1000 query 统计 Fable vs Soul 拒绝率, 量化 "Nate 主观感觉".
  3. 跨 harness 验证 — Nate 用 Claude Code 跑 Fable, Codex 跑 Soul. 同样的 prompt 换 harness 会怎样? 设计 A/B 测试.
