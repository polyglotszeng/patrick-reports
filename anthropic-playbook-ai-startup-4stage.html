AI 時代怎麼創業？Anthropic Playbook 一次看懂四階段 workflow

Gary Chen · 17:03 min · 2026-06-06
AI Startup Anthropic Claude Workflow Solo Agent

Key Takeaways

  • #1 Ten-person unicorn 的本質不是人變聰明，而是每個人背後都有 AI 系統放大產出。核心心法：「先用 AI 跑流程，再決定哪些流程值得變成職位」。
  • #2 AI 放大執行力，也放大錯誤。沒有判斷力的人，錯誤會用更快的速度被放大。以前工程成本高是創業者的煞車；現在錯誤產品可以被 AI 包裝得有模有樣。
  • #3 42% startup 失敗是因為 built something nobody wanted。AI 不會自動解決這個問題，AI 只會讓你更快做出一個沒人要的東西。
  • #4 用 Claude 做反向驗證：不要問「這個市場為什麼有機會」（它在幫你找支持證據），而是請 Claude 扮演反方，找出這個 idea 最可能失敗的原因。
  • #5 Customer discovery 問過去，不問未來。不要問「你會不會用」，要問「你上一次遇到這個問題是什麼時候，當時怎麼處理的」。
  • #6 MVP stage：AI 最擅長 greenfield（從零做 demo），不擅長 brownfield（已累積技術債的產品迭代）。Scope creep 現在沒有工程成本擋著了，「不做」會比「做」更重要。
  • #7 CLAUDE markdown 是 AI session 的工作記憶，讓 Claude Code 每次進來都知道專案的基本規則，避免 codebase 漂移。
  • #8 Launch stage：AI 不應該只是聊天機器人，應該是營運系統的一部分，把重複出現的工作流變成固定會運轉的系統。
  • #9 Scale stage 三個護城河：accumulated depth（累積深度）、proprietary system data（獨有系統資料）、workflow lock-in（工作流綁定）。
  • #10 真正的差距不是誰有沒有用 AI（很快就會不是差異），而是誰能把 AI 系統化，把 AI 變成工作系統。

Stage 1: Idea — AI 是研究員 + 反方辯論手

五個關鍵問題

痛點是否真的存在 → 誰有這個問題 → 問題發生得夠頻繁嗎 → 大家現在怎麼解決 → 解法到底有沒有打中真正的問題

三個 Use Case

① 把模糊的 idea 變成可以驗證的假設
給 Claude：你觀察到的痛點、你服務的人他們現在的工作流程、你懷疑哪個地方有摩擦。讓 Claude 幫你拆成五個問題，最後判斷這些問題是否可一一回答。

② 反向驗證（重要！）
不要問「請幫我分析這個市場為什麼有機會」（它在幫你找支持證據）。請 Claude 扮演反方：客戶為什麼可能根本沒有這個痛點、競品為什麼已經夠好、市場為什麼可能太小、解法為什麼很難賣。

③ Customer Discovery 訪談設計
問過去不問未來。Claude 可以幫你設計訪談問題、檢查哪些問題太具備誘導性、哪些問題太過抽象。

Stage 2: MVP — 用 Scope、Context、Security 管住 AI

核心問題：AI 太願意寫了

以前 scope creep 會被工程成本擋住（工程師皺眉、PM 排優先級、老闆看 budget）。現在每個突發奇想都可以很快變成產品的一部分——這就是無意義的需求蔓延。

三個 Use Case

① MVP Scope Doc
定義做什麼、不做什麼、排除功能（這塊一定要寫！）。判斷這個功能是否是能驗證產品價值的核心，還是只是手癢想加。

② CLAUDE Markdown
給未來 AI session 看的工作記憶、員工手冊。讓 Claude Code 每次進來都知道專案的基本規則，在結構上保持一致性，避免 codebase 漂移。

③ Security Review
AI 寫的程式碼能跑不等於安全。功能對不對一眼就看出來，但安全漏洞不會報錯、不會當機。先讓 Claude 掃一輪：authentication、session 處理、API 資料洩漏、injection 風險。

Stage 3: Launch — 從 Founder 手工變成系統

Launch 不只是產品上線，而是公司從 founder 手工操作變成可以重複運轉的系統。AI 不應該只是聊天機器人，應該是營運系統的一部分。

兩個 Use Case

① Weekly Metrics Brief
固定產出每週 metrics report：輸入各種產品和營運指標，AI 產出本週變化、異常訊號、可能原因、下週應該追的問題。
Sean Ellis Test：問活躍用戶「如果你不能再用這個產品，你會有多失望？」如果 >40% 回答 very disappointed = 有意義的 PMF 指標。

② Support 和 Bug Triage SOP
建立分類規則、優先級、標準回覆、升級條件。讓你知道哪些問題可以流程化，哪些代表產品本身需要改。

Stage 4: Scale — 三個護城河

護城河含義
Accumulated Depth對領域的理解、產品整合深度、別人沒有的資料和流程
Proprietary System Data產品被用得越久，越懂這群人怎麼做事，競爭對手看 landing page 抄不走
Workflow Lock-in客戶把自動化、團隊習慣、標準輸出都建在你的產品上，替換你是個 operational project

兩個具體做法

User Behavior Data Flywheel：從使用者行為找出可以持續改善產品的訊號——哪些 output 被改掉、哪些流程被重複使用、哪些功能沒人碰。持續追蹤轉成優化 idea，模型越來越準，越準越多人用。

避免只有一個功能：今天做摘要，明天別人也做摘要，沒技術壁壘。但如果產品已經接到他的資料來源進入他的團隊流程，影響他的標準輸出——離開成本就高到不像話。
