Key Takeaways (10 条)

  1. Phrase #1 launch sub-agents — 强制并行 Claude 会话, 突破"主 agent 倾向独干"的默认。3 场景: 多视角 (5 agent 同题异判, 避免 anchor convergence) / 开新可能 (10 agent 扫 10K domains) / 独立任务并行提速。
  2. Phrase #2 write me an implementation spec — Boris Cherny 原话: 建任何东西前先建 spec/plan。Karpathy 升级: plan mode 不够, 要协作"写 spec"而非"通过 plan mode"。5×5×5 = 125 组合 vs 1×1×1 = 1 组合的统计论证。
  3. Phrase #3 interview me — 反向流程, 让 Claude 问你"你不知道要问的"问题, 5 分钟从空白页到完整 spec。三要素: ① identify core problem + 是/不是给谁 ② walk through key decisions ③ summarize back as spec。陷阱: 急答 yes/no = 跟 auto-answer 等价。
  4. Phrase #4 verify before you build — Boris 原话: 给 Claude 反馈环 → 质量 2-3x。再加一层: 给 Claude 工具看自己 output → 自纠正。三层法: ① Claude.md 加 1 行 "before work, mention how to verify" ② 启 verification tools (external/internal, tech/non-tech) ③ 标识 human validation zones (high cost of error 需人 sign off)。
  5. Phrase #5 based on this conversation, build me a skill — 反主流"先想 skill 名, 再实现"路径, 强制从已完成对话逆向生成 skill (使用场景已验证)。Anthropic 内部增强法: 加 gotchas section (edge case / stylistic quirk / 来回修过的点), "enhance any skill to include gotchas" 闭环。
  6. Phrase #6 automate this (cautionary) — 是 6 phrase 里最危险的一个。Taste test: 需"品味判断"输出 → 只能 augment 不能 automate; 80/20 analysis: 80% 合格就 automate, 否则 augment。Boris 标 3 个最强 automation features: hook / schedule / loops。警钟: 99% 人不跑这 2 个 filter, 制造一堆"waste tokens + AI slop at scale"。
  7. 6 phrase 整合闭环 (从 0 到自动化系统): sub-agents (并行) → spec (对齐) → interview (抽需求) → verify (反馈环) → skill (固化重复) → automate (小心 80/20 + taste test)。顺序很重要, 反过来失败率高。
  8. Gotchas section 是 skill 质量的杠杆 — Anthropic 内部文章强调, "when first make skill → not perfect → small issues = gotchas → include them → don't make same mistake twice"。反馈环越短, skill 进化越快。Concrete beats abstract every time。
  9. "Should I turn this process into a skill?" 优于 "What skill should I create?" — 从"先抽象设计"翻转为"先做再固化", 跟 Patrick 偏好的"3 步闭环" (Mapping → 选 1 → 立即建) 同一精神。
  10. Human validation zones 三维度判定: cost of error (高=human) × frequency (低=human) × reversibility (不可逆=human)。80/20 版 Claude.md 模板: "changes touching [X zone] require human sign-off"。

Section-by-Section Summary

Part 1: 0:00-0:26 — Setup & Misconception

"Austin says 6 phrases 才是 unlock, 不是更好 prompt / 更聪明模型 / 装更多 plugin"。卖点: "exact thing to type into Claude" + "cautionary phrase that 99% people get wrong"。这是典型 clickbait 钩子, 但 phrase 本身有结构, 跟 3 红旗警告相符。

Part 2: 0:26-3:14 — Phrase #1: launch sub-agents

核心论点: Claude 默认 underutilizes sub-agents (业务原因 + "main agent 倾向自己干"), 显式 launch sub-agents 强制分拆。3 场景:

  • 多视角: 5 agent 同题异判 (避免 anchor convergence — 同 chat 5 次会逐次收束)
  • 开新可能: 10 agent 扫 10K domains (例: BuildPartner.ai 域名, "would never do if single agent")
  • 独立任务并行: 提 80/20 速

注意事项: 子 agent 数量 ≠ 效果, crystal clear task 是前提 (Austin 自警)。

Part 3: 3:14-5:45 — Phrase #2: write me an implementation spec

Boris Cherny 原话引述: "before you build anything, you need to create a plan or a spec"。Karpathy 升级: "I don't even like the plan mode... you have to work with your agent to design a spec that is very detailed, basically the docs, and then get the agents to write them."

统计论证: 3 步 × 5 选项 = 125 组合 vs spec 后 1×1×1 = 1 组合。"literally doesn't matter how good these models get. It doesn't know what you don't tell it."

实战 prompt 关键行: for each step, show me the key decisions you'd make (逼 Claude 拆解, 让 user 提前 override)

Part 4: 5:45-7:52 — Phrase #3: interview me

反向流程: Claude 问你, 不是你问 Claude。3 关键 meta:

  1. work with me to identify the core problem we're solving, who it is and isn't for (Anthropic 自身清晰定位)
  2. work through any key decisions together (按 step 答, 不会的答 "use your best judgment" = 知情 yolo)
  3. summarize it back to me as an implementation spec (从空白页到 spec < 5 min)

陷阱: 急答 yes/no = 等价 auto-answer, 失去价值。Pro tip: 用 Wispr Flow / Hex 录音, 模拟真面试。

Part 5: 7:52-11:09 — Phrase #4: verify before you build

Boris 原话: "Give Claude a way to verify its work. If it has a feedback loop, it will 2-3x the quality"。Boris 升级: "give Claude a tool to see the output of its work" → 自纠正。3 层架构:

  • L1 Claude.md 一行: Before you do any work, mention how you could verify that work (因为 Claude.md 注入每个 prompt)
  • L2 启工具: 问 Claude "based on what I'm building, what are some tools that could help with verification?" 例: Hostinger MCP 验 deploy / brand-voice skill 验文案
  • L3 Human validation zones: 找 cost of error 高的部分 (例: payment 系统), 强制 sign off; cost of error 低 (例: 落地页 UI) → 接受 build fast break things

80/20 集成 prompt (Austin 提供): 一段话覆盖三层

Part 6: 11:09-12:59 — Phrase #5: based on this conversation, build me a skill

核心反主流: 主流"先想 skill 名 → 再实现"导致两种失败 (想不出 / 不用)。Austin 翻转: skill 只能从已完成对话抽取, 使用场景已验证。Anthropic 内部 gotchas section: "small issues that come up = gotchas = edge cases / stylistic quirks / 来回修过的点"。反 prompt: Based on this conversation, enhance any skill I use to include a gotchas section (闭环: 用了就升级)。金句: "Should I turn this process into a skill?" 优于 "What skill should I create?"

Part 7: 12:59-15:12 — Phrase #6: automate this (the dangerous one)

Austin 自警: "this power phrase is one that I'm the most skeptical of"。2 决策 filter:

  1. Taste test: 任务需"品味"判断好/坏输出? → 只能 augment。完全可量化? → 可考虑 automate。
  2. 80/20 output analysis: 输出 80% 合格我接受吗? → accept = automate, reject = augment。

Boris 3 最强 features: hook / schedule / loops。警告: 99% 人不跑 filter, 制造 "waste tokens + AI slop at scale"。
